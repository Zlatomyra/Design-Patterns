# Design-Patterns
Лабораторна робота №1: Design Patterns

Мета: ознайомитися з основними шаблонами проєктування та показати, як їх можна застосовувати на практиці, використовуючи приклад системи замовлення кави та напоїв.

Використані патерни:
Factory Method

Проблема: У системі потрібно створювати різні види напоїв, наприклад каву, чай або лате, але ми не хочемо, щоб клієнтський код був прив’язаний до конкретних класів. Наприклад, якщо код підтримує лише каву і весь код працює тільки з класом кави, то код не зможе працювати з іншими напоями. Кожного разу, коли ви захочете додати новий напій, прийдеться міняти код.

Ідея: Створюємо абстрактну фабрику з методом для створення напою. Конкретні підкласи реалізують цей метод і повертають потрібний напій. Клас-фабрика знає, який конкретний напій створювати, а клієнтський код просто звертається до фабрики. Таким чином легко додавати нові типи напоїв без зміни існуючого коду.

Bridge Method

Проблема:
Наприклад, у вас є система для замовлення напоїв і спочатку всі напої подаються тільки в чашках. З часом виникає потреба подавати напої також у пляшках або склянках. Якщо код напою безпосередньо відповідає за упаковку, додавання нових способів подачі змушує змінювати весь код напоїв, і це швидко стає громіздко та незручно.

Ідея:
Ми розділяємо напій і реалізацію упаковку на окремі класи. Напій зберігає посилання на об’єкт-пакувальник, який відповідає за спосіб подачі. Таким чином можна комбінувати будь-який напій з будь-яким типом упаковки без зміни існуючих класів. Нові напої чи нові пакувальники додаються легко і незалежно один від одного.

Decorator Method

Проблема:
Потрібно додавати нові інгредієнти до напою, наприклад молоко, сироп або спеції. Якщо ми будемо змінювати код базового класу напою для кожного нового інгредієнта, програма швидко стане громіздкою і важкою для підтримки.

Ідея:
Використовуємо декоратори — окремі класи, які обгортають базовий напій і додають нову поведінку або властивості. Наприклад, збільшують вартість або додають інгредієнт. Кілька декораторів можна комбінувати між собою, створюючи будь-які комбінації інгредієнтів без зміни коду основного класу напою.
